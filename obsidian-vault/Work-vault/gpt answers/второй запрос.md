# 1. Общая архитектура приложения

Приложение выглядит как Android клиент портативного ультразвукового сканера. Основная задача приложения:

1. Получить кадры от датчика.
2. Распарсить их в Java объект кадра.
3. Выполнить scan conversion.
4. Преобразовать grayscale данные в bitmap.
5. Наложить Doppler слои.
6. Нарисовать измерительные элементы, шкалы, ROI, PW gate, M mode, needle overlay.
7. При необходимости выполнить дополнительные анализы:
   8. B line detection.
   9. PW envelope analysis.
   10. Vessel direction analysis.
   11. Needle position calculation.

Упрощённая схема:

```text
Датчик
↓
Пакеты данных
↓
Парсер протокола
↓

↓
b_data, color_data, power_data, pw_data, miscs
↓

↓

↓

↓

↓

```


Внутри `f8789f` лежат разные данные:

```text
"miscs"       содержит дополнительные данные, например needle_data
```

# 2. B mode: что это физически

B mode это обычное серошкальное ультразвуковое изображение. Датчик посылает короткий акустический импульс, затем слушает отражённый сигнал.

Физика базовая:

```text
датчик испускает импульс
импульс идёт в ткань
на границах тканей часть энергии отражается
эхо возвращается обратно
по времени задержки вычисляется глубина
по амплитуде эха вычисляется яркость пикселя
```

Основная формула глубины:

```text
d = c t / 2
```

Где:

```text
d  глубина отражателя
c  скорость звука в ткани
t  время от отправки импульса до приёма эха
2  потому что сигнал прошёл путь туда и обратно
```

Скорость звука в мягких тканях обычно принимается:

```text
c = 1540 м/с
```

В сантиметрах в секунду:

```text
c = 154000 см/с
```

Половина скорости:

```text
c / 2 = 770 м/с
```

Именно поэтому в коде видна константа вида:

```java
770000.0 / sampleRate
```

Это почти наверняка преобразование индекса сэмпла в физическую глубину.

Если частота дискретизации задана в согласованных единицах, то расстояние между двумя соседними сэмплами по глубине:

```text
Δd = c / (2 Fs)
```

Где:

```text
Fs  частота дискретизации
```

В коде это выглядит примерно так:

```java
sampleSpacing = 770000.0f / f6;
```

То есть:

```text
номер сэмпла y
↓
глубина = y · sampleSpacing
```

# 4. Почему нужен scan conversion

Сырые ультразвуковые данные обычно лежат не в экранных координатах.

Один столбец данных это один ультразвуковой луч:

```text
x  номер луча
y  глубина вдоль луча
```

Но экранная картинка это обычная декартова система:

```text
screenX
screenY
```

Поэтому нужно преобразовать:

```text
луч, глубина
↓
физическая точка в секторе
↓
экранный пиксель
```

Это и называется scan conversion.

# 5. Геометрия датчиков

В приложении видна универсальная секторная модель.

Есть параметры:

```text
R       радиус кривизны
Φ       полный угол сектора
N       число лучей
M       число сэмплов по глубине
Δd      физический шаг по глубине
```

Для convex датчика радиус конечный.

Для phased array радиус может быть близок к нулю или использоваться особая модель.

Для linear датчика приложение использует хитрый приём: делает огромный радиус.

В коде:

```java
public static final float f8838a = 100000.0f;

public static float a(float f4) {
    return (float) (Math.asin((((double) f4) / 2.0d) / 100000.0d) * 2.0d);
}
```

Смысл:

```text
linear датчик можно представить как сектор с гигантским радиусом
```

Если радиус огромный, дуга почти плоская. Тогда секторная геометрия становится почти прямоугольной.

Формула угла для линейного датчика:

```text
Φ = 2 arcsin(W / (2R))
```

Где:

```text
W  физическая ширина апертуры
R  огромный радиус, например 100000
```

При большом R:

```text
sin(x) ≈ x
```

Поэтому:

```text
Φ ≈ W / R
```

И дуга радиуса R с углом Φ имеет длину:

```text
R Φ ≈ W
```

То есть получается почти плоская линейная апертура.

# 6. Прямое преобразование координат

Допустим есть точка в сырой сетке:

```text
i  индекс луча
j  индекс сэмпла по глубине
```

Тогда физическая глубина:

```text
d = j · Δd
```

Расстояние от центра кривизны:

```text
ρ = R + d
```

Угол луча:

```text
θ = Φ · (i - (N - 1) / 2) / (N - 1)
```

То есть центральный луч имеет угол примерно:

```text
θ = 0
```

Левый край:

```text
θ = -Φ / 2
```

Правый край:

```text
θ = +Φ / 2
```

Переход в физические декартовы координаты:

```text
X = ρ sin θ
Y = ρ cos θ
```

Если нужно вычесть базовый радиус, можно использовать:

```text
Y = ρ cos θ - R
```

Но в коде удобнее держать origin так, чтобы экранная система уже учитывала смещение.

Далее физические координаты переводятся в пиксели:

```text
screenX = X / pixelSpacing + originX
screenY = Y / pixelSpacing + originY
```

Где:

```text
pixelSpacing  физический размер одного экранного пикселя в выбранной bitmap
originX       положение центра сектора на bitmap
originY       положение центра сектора на bitmap
```

В коде `m2.r.a` логика похожа именно на это:

```java
double d4 = pointF.x;
double d5 = pointF.y * this.f8861p;
float radius = this.f8858m;
double r = d5 + radius;
```

То есть:

```text
sampleIndex → physicalDepth
physicalDepth + radius → radialDistance
```

Дальше:

```java
double x = Math.sin(angle) * r;
double y = Math.cos(angle) * r;
```

И перевод в пиксели.

# 7. Обратное преобразование координат

Обратное преобразование нужно для касаний и интерактивных элементов.

Например пользователь пальцем двигает Color Doppler box. Палец даёт экранные координаты:

```text
screenX, screenY
```

Но аппаратные параметры нужно отправлять в координатах исходной сетки:

```text
beamIndex, sampleIndex
```

Обратное преобразование:

```text
X = (screenX - originX) · pixelSpacing
Y = (screenY - originY) · pixelSpacing
```

Далее:

```text
ρ = sqrt(X² + Y²)
```

Угол:

```text
θ = atan2(X, Y)
```

Почему `atan2(X, Y)`, а не `atan2(Y, X)`?

Потому что в этой геометрии:

```text
X = ρ sin θ
Y = ρ cos θ
```

Значит:

```text
tan θ = X / Y
```

Поэтому:

```text
θ = atan2(X, Y)
```

Глубина:

```text
d = ρ - R
```

Индекс сэмпла:

```text
j = d / Δd
```

Индекс луча:

```text
i = θ · (N - 1) / Φ + (N - 1) / 2
```

Именно это нужно для:

```text
перемещения Doppler ROI
перемещения PW gate
измерительных курсоров
needle overlay
vessel ROI
B line mapping
```

# 10. Grayscale LUT

B mode данные изначально лежат как `byte`, то есть примерно:

```text
0..255
```

Но напрямую рисовать эти значения не всегда хорошо. Нужно применить кривую яркости, контраст, gamma, dynamic range.

В `m2.h` видны LUT.

Пример gamma преобразования:

```text
x = input / 255
y = clamp((x - offset)^γ, 0, 1)
output = 255 y
```

Формула:

```text
Iout = 255 · clamp((Iin / 255 - 0.05)^γ, 0, 1)
```

Где:

```text
γ меньше 1 делает тёмные области ярче
γ больше 1 делает тёмные области темнее
```

Также есть sigmoid LUT:

```text
y = low + (high - low) / (1 + exp(-k (x - center)))
```

Это S образная кривая.

Она полезна для контраста:

```text
тёмный шум прижимается вниз
средние структуры усиливаются
яркие области насыщаются
```

# 11. Color Doppler

Color Doppler показывает направление и скорость движения крови.

Физика Doppler:

Если отражатель движется, частота отражённого сигнала меняется.

Основная формула:

```text
fd = 2 f0 v cos α / c
```

Где:

```text
fd  Doppler shift
f0  частота ультразвука
v   скорость крови
α   угол между лучом и направлением потока
c   скорость звука
```

Отсюда скорость:

```text
v = fd c / (2 f0 cos α)
```

Color Doppler обычно показывает не полный спектр, а усреднённую скорость в ROI.

В классе `m2.y`:

```java
public class y {
    public byte f8915b;
    public byte f8916c;
    public int f8917d;
    public int f8918e;
    public int f8919f;
    public int f8920g;
    public float f8921h;
    public int f8922i;
    public float f8923j;
}
```

Вероятная интерпретация:

```text
f8915b  velocity или phase карта
f8916c  alpha mask или дополнительная карта
f8917d  left ROI
f8918e  right ROI
f8919f  top ROI
f8920g  bottom ROI
f8921h  steer angle
f8923j  focus markers
```

В palette коде видно:

```java
if (value != 0) {
    if (v >= 0) {
        positivePalette
    } else {
        negativePalette
    }
}
```

То есть Color Doppler значение знаковое:

```text
v > 0  поток в одну сторону
v < 0  поток в другую сторону
v = 0  нет валидного цветового сигнала
```

Обычно:

```text
к датчику     красный
от датчика    синий
```

Но это может инвертироваться настройкой.

# 12. Power Doppler

Power Doppler показывает не направление, а энергию Doppler сигнала.

Физический смысл:

```text
Color Doppler: скорость со знаком
Power Doppler: сила движения без знака
```

Если Color Doppler отвечает:

```text
куда и насколько быстро течёт кровь
```

То Power Doppler отвечает:

```text
насколько уверенно здесь есть кровоток
```

Класс `m2.c0`:

```java
public class c0 {
    public byte f8728b;
    public byte f8729c;
    public int f8730d;
    public int f8731e;
    public int f8732f;
    public int f8733g;
    public float f8734h;
    public float f8736j;
}
```

Поля похожи на Color Doppler:

```text
f8728b  power карта
f8729c  alpha mask или доп карта
f8730d  left ROI
f8731e  right ROI
f8732f  top ROI
f8733g  bottom ROI
f8734h  steer angle
f8736j  focus markers
```

Power Doppler palette не использует знак:

```text
power = 0..255
цвет = palette
```

# 13. Steer angle

В Color и Power есть угол steering:

```text
Color: f8921h
Power: f8734h
```

Это наклон Doppler лучей. Для линейного датчика часто можно наклонять лучи, чтобы улучшить угол к сосуду.

Если Doppler box наклонён, координаты внутри ROI поворачиваются.

Обычная матрица поворота:

```text
x' = cos β · x - sin β · y
y' = sin β · x + cos β · y
```

Где:

```text
β  steer angle
```

В коде есть методы типа:

```text
J(point, angle)
B(point, angle)
```

Они делают прямое и обратное преобразование с учётом наклона.

При наклоне итоговая область может стать шире:

```text
newWidth = baseWidth + 2 · height · sin(|β|)
```

Смысл:

```text
наклонённый прямоугольник занимает больше места по горизонтали
```

# 14. ROI Color и Power

ROI задаётся в координатах сырой сетки:

```text
left
right
top
bottom
```

Например для Color:

```text
left   = f8917d
right  = f8918e
top    = f8919f
bottom = f8920g
```

Чтобы нарисовать его на экране, берутся четыре угла:

```text
(left, top)
(right, top)
(left, bottom)
(right, bottom)
```

Каждая точка проходит:

```text
raw coordinate
↓
scan conversion
↓
steer rotation
↓
view scaling
↓
screen coordinate
```

Если пользователь двигает ROI пальцем, используется обратная цепочка:

```text
screen coordinate
↓
inverse view scaling
↓
inverse scan conversion
↓
raw coordinate
```

Так приложение вычисляет, на сколько сдвинуть ROI в координатах датчика.

# 15. PW Doppler

PW Doppler это Pulsed Wave Doppler. Он показывает спектр скоростей во времени из маленькой области, называемой sample volume или gate.

Принцип:

```text
выбирается один луч
на этом луче выбирается глубина gate
из этой области собирается Doppler сигнал
делается FFT по времени
получается спектр частот
частоты переводятся в скорости
```

```java
    public int f8718i;

    public int f8720k;
    public int f8721l;
    public byte f8722m;
    public float f8723n;
    public byte f8724o;
    public float f8725p;
    public byte f8726q;
}
```

Вероятная интерпретация:

```text
f8711b  steer angle или угол линии
```

# 16. Физика PW скорости

Формула Doppler:

```text
fd = 2 f0 v cos α / c
```

Отсюда:

```text
v = fd c / (2 f0 cos α)
```

В коде видна формула:

```java
D =
    (((int) h.p(b0Var.f8718i)) * 154000.0f)
    /
    (((b0Var.f8719j * 1000000.0f) * 4.0f)
    * cos(angleCorrection));
```

Сравним с теорией:

```text
v = fd c / (2 f0 cos α)
```

В коде:

```text
c = 154000 см/с
f0 = f8719j · 1000000 Гц
cos α = cos(angleCorrection)
```

`h.p(f8718i)` вероятно даёт частотный масштаб, связанный с PRF или Nyquist частотой.

Почему в знаменателе `4`, а не `2`?

Возможные причины:

1. Внутренний параметр `h.p()` может быть не полным Doppler shift, а величиной, уже связанной с удвоенной шкалой.
2. Спектр может быть нормирован относительно половины PRF.
3. FFT картинка может иметь диапазон `±PRF/4` в пиксельной шкале.
4. Используется IQ демодуляция и внутренняя шкала делится иначе.

Но структура физически очевидна:

```text
скорость ∝ c
скорость ∝ Doppler frequency scale
скорость обратно пропорциональна f0
скорость обратно пропорциональна cos угла
```

Чем хуже угол, тем больше correction:

```text
если α = 0°, cos α = 1
если α = 60°, cos α = 0.5, скорость удваивается
если α → 90°, cos α → 0, расчёт становится нестабильным
```

Поэтому в реальном УЗИ стараются держать Doppler angle:

```text
≤ 60°
```

# 17. PW gate на B mode

PW gate задаётся:

```text
x = f8712c
```

То есть:

```text
выбран один луч
```


Если есть angle correction, рисуется дополнительная линия, показывающая предполагаемое направление сосуда.

Поворот выполняется стандартно:

```text

```

Где:

```text
(cx, cy)  центр поворота
α         угол
```

# 18. PW spectrum image

PW spectrum обычно рисуется снизу экрана.

В коде видно деление экрана:

```text
верхние 70 процентов  B mode
нижние 30 процентов   PW spectrum
```

Спектр это картинка:

```text
x  время
y  Doppler frequency или velocity
```

Baseline обычно посередине:

```text
выше baseline  положительные скорости
ниже baseline  отрицательные скорости
```

Если высота спектра `H`, то:

```text
baselineY = H / 2
```

Но в коде baseline может сдвигаться:

```text
baselineY = q · H / 4
```

Где `q` это параметр baseline shift.

Перевод пикселя в скорость:

```text
v(y) = (baselineY - y) · velocityMax / (H / 2)
```

Если ось вниз положительная, то:

```text
v(y) = (baselineY - y) · scale
```

Где:

```text
scale = velocityMax / (H / 2)
```

## 19.2 Предобработка спектра

В `GenCurve` видно:

```text
medianFilter(3,5)
meanFilter(5,5)
dilation(5,5)
erosion(5,5)
```

Что это значит.

### Median filter

Медианный фильтр заменяет пиксель медианой соседей.

Он хорошо удаляет одиночный шум:

```text
0 0 0
0 255 0
0 0 0
```

После медианы яркая точка исчезнет.

### Mean filter

Средний фильтр сглаживает изображение:

```text
новый пиксель = среднее соседей
```

Формула:

```text
I'(x,y) = 1 / (mn) · Σ I(x+i, y+j)
```

### Dilation

Дилатация расширяет яркие области:

```text
I'(x,y) = max I(x+i, y+j)
```

### Erosion

Эрозия сужает яркие области:

```text
I'(x,y) = min I(x+i, y+j)
```

Последовательность:

```text
dilation затем erosion
```

Это morphological closing.

Она:

```text
закрывает маленькие дырки
соединяет близкие яркие области
делает спектр более цельным
```

## 19.3 Fuzzy C Means threshold

В коде видны вызовы fuzzy C means с параметрами вроде:

```text
0.6
0.2
```

Fuzzy C Means это мягкая кластеризация.

Обычная идея:

Есть пиксели с яркостью:

```text
x_i
```

Нужно разделить их на классы:

```text
фон
сигнал
```

Но не жёстко, а через принадлежность:

```text
u_ik ∈ 
```

Где:

```text
u_ik  степень принадлежности пикселя i к классу k
```

Целевая функция FCM:

```text
J = Σ_i Σ_k u_ik^m · ||x_i - c_k||²
```

Где:

```text
c_k  центр кластера
m    fuzziness parameter, обычно около 2
```

Обновление центров:

```text
c_k = Σ_i u_ik^m x_i / Σ_i u_ik^m
```

Обновление принадлежности:

```text
u_ik = 1 / Σ_j ( ||x_i - c_k|| / ||x_i - c_j|| )^(2/(m-1))
```

В контексте PW это нужно, чтобы отделить:

```text
яркий спектр кровотока
от
тёмного фона и шума
```

## 19.4 Выбор направления кривой

Спектр может быть выше baseline или ниже baseline.

Код считает:

```text
POS DOT
NEG DOT
POS AREA
NEG AREA
```

То есть:

```text
сколько ярких точек выше baseline
сколько ярких точек ниже baseline
какая площадь сигнала выше
какая площадь сигнала ниже
```

Если больше сигнала сверху, envelope ищется сверху.

Если больше сигнала снизу, envelope ищется снизу.

Если сигнала слишком мало:

```text

```

## 19.5 Построение envelope

Для каждого столбца времени `x` алгоритм ищет границу яркой области.

Если поток сверху baseline:

```text
идём от верхней части к baseline
ищем первый или самый дальний значимый яркий сегмент
получаем y(x)
```

Если поток снизу:

```text
идём от нижней части к baseline
ищем значимый яркий сегмент
получаем y(x)
```

Результат:

```text
curve = y
```

Затем кривая сглаживается:

```text
median filter по curve
moving average
```

Например moving average на 5 точек:

```text
curveSmooth =
(curve + curve + curve + curve + curve) / 5
```

## 19.6 FindPeak

`FindPeak` берёт curve и baseline.

Амплитуда скорости в пикселях:

```text
A(x) = |curve - baseline|
```

Пик это локальный максимум `A(x)`.

Условие локального максимума:

```text
A(x) > A(x-1)
A(x) ≥ A(x+1)
A(x) достаточно большой
```

Алгоритм группирует соседние точки, чтобы не считать один широкий пик много раз.

В результате возвращаются последние или наиболее значимые три пика.

## 19.7 FindSD

По коду видно, что поиск начинается около:

И идёт назад по кривой, анализируя наклон:


В медицинском смысле это может быть:

# 28. Projection иглы на УЗИ плоскость

Игла в 3D может задаваться двумя точками:

```text
p1 = (x1, y1, z1)
p2 = (x2, y2, z2)
```

Линия иглы:

```text
p(λ) = p1 + λ (p2 - p1)
```

Где:

```text
λ  параметр вдоль иглы
```

УЗИ плоскость можно считать:

```text
Y = 0
```

Точка пересечения с плоскостью:

```text
p_y(λ) = y1 + λ (y2 - y1) = 0
```

Отсюда:

```text
λ = -y1 / (y2 - y1)
```

Если `λ` попадает в разумный диапазон, игла пересекает плоскость изображения.

Проекция на экран:

```text
screenX = originX + x / scale
screenY = originY + z / scale
```

Если приложение рисует линию по `x,z` без пересечения, значит оно показывает ортогональную проекцию иглы на плоскость УЗИ.

# 29. Out of plane indicator

Java код использует `y` для мини индикатора справа.

Это логично:

```text
Y показывает расстояние иглы от плоскости сканирования
```

Можно рисовать индикатор:

```text
Y > 0  игла перед плоскостью
Y < 0  игла за плоскостью
Y = 0  игла в плоскости
```

Математика расстояния до плоскости:

Если плоскость:

```text
n · p + d = 0
```

Тогда расстояние:

```text
dist = (n · p + d) / ||n||
```

Для плоскости `Y = 0`:

```text
n = (0,1,0)
d = 0
dist = y
```

## 30.5 Magnetic dipole model

Если это магнитная навигация, может быть модель магнитного диполя.

Поле магнитного диполя:

```text
B(r) = μ0 / (4π) · ( 3 r (m · r) / ||r||⁵ - m / ||r||³ )
```

Где:

```text
B  магнитное поле
r  вектор от магнита до сенсора
m  магнитный момент
```

Если библиотека оценивает положение магнита по нескольким магнитным датчикам, может решаться обратная задача:

```text
найти r и m, чтобы рассчитанное B было близко к измеренному B
```

Минимизируется ошибка:

```text
E(r,m) = Σ_i ||B_measured_i - B_model_i(r,m)||²
```

Это может решаться итерационно.

Признаки такого алгоритма:

```text
много sqrt
pow(distance, 3)
pow(distance, 5)
итерационный цикл
градиент
least squares
```

Это могут быть transform matrices.

Это могут быть quaternion плюс translation.

Например:

# 32. Взаимодействие needle с scan conversion

Важно понять: needle overlay не проходит обычный sector scan conversion как B mode.

B mode точка задаётся как:

```text
beamIndex, sampleIndex
```

Needle точка задаётся физически:

```text
x, y, z
```

Для отображения используется масштаб из scan geometry:

```java
PointF origin = zVar.f(new PointF(0, 0));
float scale = zVar.A();
```

И потом:

```text
screenX = originX + x / scale
screenY = originY + z / scale
```

То есть игла уже в физических координатах плоскости изображения.

Это говорит о том, что `libneedle_location.so` скорее всего возвращает координаты уже в системе УЗИ датчика, а не в координатах пикселей.

# 35. Главный вывод

По текущей картине приложение делает следующее:

```text
B mode строится из лучей и глубин через секторную геометрию
глубина считается из времени пролёта с использованием c/2
linear датчик моделируется как сектор с огромным радиусом
Color Doppler кодирует знаковую скорость
Power Doppler кодирует энергию движения без знака
PW Doppler строит спектр скоростей во времени
PW velocity scale основан на Doppler equation
M mode берёт один B mode луч из каждого кадра
VesselDirAna находит направление сосуда через segmentation и edge detection
VFCalcCore строит envelope PW спектра и ищет пики
Needle navigation переводит внешние 3D данные иглы в плоскость УЗИ изображения
```

Самые важные физические формулы:

```text
Глубина:
d = c t / 2

Шаг глубины:
Δd = c / (2Fs)

Секторная геометрия:
θ = Φ(i - (N-1)/2)/(N-1)
X = (R + jΔd) sin θ
Y = (R + jΔd) cos θ

Doppler shift:
fd = 2f0vcosα / c

Скорость:
v = fdc / (2f0cosα)

Angle correction:
vcorrected = vmeasured / cosα

3D transform иглы:
p_image = T_image_probe · T_probe_sensor · T_sensor_needle · p_needle

Projection иглы:
screenX = originX + X / scale
screenY = originY + Z / scale
outOfPlane = Y
```
